# Shinka Repo Scan Skill

`shinka-scan-repo` explores a repository, identifies functions that would
benefit from ShinkaEvolve, and produces a ranked candidate report.

This skill is **read-only**. It does not modify the repo, does not start
evolution, and does not write `EVOLVE-BLOCK` markers into source files.

## Composition With Existing Shinka Skills

The core ShinkaEvolve skills live in the `skills/` directory of the ShinkaEvolve repo. If the skills are not already loaded by the agent, first check whether a source clone of ShinkaEvolve exists in the current project or directory tree. If not found, install via `uv pip install shinka-evolve` (prefer `uv` for package management unless the user or project explicitly uses something else).

This skill is a **triage front-end** for those native skills. It does not set up tasks itself, does not start
evolution, and does not write `EVOLVE-BLOCK` markers. It only proposes
candidates and then hands the user-confirmed choice to a native skill:

- `shinka-convert` ŌĆö preferred when adapting an existing function in-place
  into a Shinka task (the function already lives in the repo).
- `shinka-setup` ŌĆö when the user wants a clean scaffold rather than adapting
  the existing file.
- `shinka-run` and `shinka-inspect` ŌĆö used downstream once the task exists.

Domain-specific peer setup layers (e.g. `shinka-create-figure`,
`shinka-self-play`) may be recommended **only** if a candidate matches their
domain pattern; otherwise default to `shinka-convert` / `shinka-setup`.

## When To Use

Trigger phrases:

- "find candidates for Shinka evolution"
- "scan this repo for Shinka"
- "what could ShinkaEvolve improve here"
- "triage this codebase for Shinka evolutionary search"
- "which functions are worth evolving with ShinkaEvolve"

Do NOT use for: code review, refactoring, full repo analysis, or non-Python
projects.

## Inputs

- Path to the repo (defaults to current working directory).
- Optional include/exclude globs.
- Optional max candidates to report (default 20).
- Optional output directory (default: `./shinka_scan_report/`).

## High-Level Procedure

- **Inventory** ŌĆö find Python files in scope. Skip binary, generated,
   vendored, and ignored paths (see "Skip paths" below).
- **Parse** ŌĆö for each file, parse the AST and extract every top-level
   function and method.
- **Atomicity filter** ŌĆö discard anything Shinka cannot evolve safely. See
   "Atomicity filter" below.
- **Score** survivors against the criteria in
   `references/candidate-criteria.md`.
- **Pattern-match** against domain templates in
   `references/domain-patterns.md` (sampler, optimizer, kernel, scheduler,
   loss, scoring, heuristic, etc.). A pattern match raises confidence and
   lets the report propose specific metrics.
- **Write** the JSON and Markdown reports matching the schema in
   `references/report-schema.md`.
- **Summarize** for the user: print the top 5 candidates inline, give the
   path to the full report.

## Skip Paths

Always skip:

- `.git/`, `.venv/`, `venv/`, `env/`, `node_modules/`, `dist/`, `build/`,
  `__pycache__/`, `.mypy_cache/`, `.pytest_cache/`, `.ruff_cache/`
- vendored copies under `third_party/`, `vendor/`, `_vendor/`
- generated code (`*_pb2.py`, `*_pb2_grpc.py`, files containing
  `# AUTOGENERATED` / `# DO NOT EDIT`)
- test files (`test_*.py`, `*_test.py`, anything under `tests/` unless the
  user explicitly opts in)

Honor `.gitignore` if present.

## Atomicity Filter

Discard a function **before scoring** if any of these hold. These are hard
exclusions, not soft penalties ŌĆö the Shinka contract cannot safely express
them.

- The function spans multiple files (or its required helpers do).
- The function relies on framework lifecycle (Django views, FastAPI routes,
  pytest fixtures, Flask app context, Click commands as entry points).
- The function performs uncontrolled I/O: network calls, database writes,
  filesystem mutation outside `/tmp`, subprocess execution against external
  systems.
- The function is a thin wrapper around a single library call (e.g.
  `return scipy.optimize.minimize(...)` with no real logic).
- The function is generated, vendored, or in a third-party copy.
- The function is decorated with anything that fundamentally changes its
  callable semantics: `@property`, `@cached_property`, `@contextmanager`,
  FastAPI / Flask / Click route decorators.

Decorators that are FINE (they preserve the semantics Shinka needs):

- `@jit`, `@njit`, `@torch.compile`
- `@lru_cache`
- `@staticmethod`, `@classmethod`
- `@dataclass`

## Scoring Criteria (summary)

See `references/candidate-criteria.md` for the full rubric. Scored axes:

- **Self-containedness** ŌĆö function + its trivially copyable helpers fit in
  one file.
- **Determinism** ŌĆö given the same inputs, it should be possible to run
  evaluations repeatably.
- **Numeric / measurable output** ŌĆö there is something obvious to score
  (loss, runtime, accuracy, count, ratio, distance).
- **Improvability surface** ŌĆö the body has algorithmic substance (loops,
  branching, math, heuristics) ŌĆö not just a pass-through.
- **Eval cost** ŌĆö single-run evaluation is cheap enough that hundreds of
  generations are feasible.
- **Domain pattern match** ŌĆö bonus when the function looks like a known
  ShinkaEvolve-friendly archetype (sampler, optimizer step, kernel,
  scheduler, loss, scoring, heuristic).

A combined score of Ōēź0.6 on a 0ŌĆō1 scale is the recommended cut-off.

## Report Outputs

Two files in the output directory:

- `report.json` ŌĆö full machine-readable report (schema:
  `references/report-schema.md`).
- `report.md` ŌĆö human-readable summary with the top candidates, file/line
  citations, proposed metrics, and the cost band.

## Workflow

1. Read user inputs (repo path, scope, output dir).
2. Inventory Python files; build an in-memory list of candidate functions.
3. Apply atomicity filter.
4. Score survivors; pattern-match.
5. Estimate cost for the top N survivors.
6. Write `report.json` and `report.md`.
7. **Present** the top 5 candidates inline with: file:line citation,
   combined score, domain pattern, suggested metrics, cost band, and a
   one-line rationale. Tell the user where the full report lives.
8. **Wait for user confirmation** of which candidate(s) to pursue. Do not
   start any setup automatically. Acceptable user responses include picking
   one candidate by index, picking several, asking for re-ranking, or
   declining (in which case stop).
9. **Hand off** the confirmed candidate(s) to a native ShinkaEvolve skill:
    - default: `shinka-convert` with the candidate's file path and qualname,
      so the existing function is adapted in-place;
    - if the user prefers a fresh scaffold: `shinka-setup`;
    - if `domain_pattern == figure-generation` and the user agrees:
      `shinka-create-figure`;
    - if `domain_pattern == game-agent-policy` and the user agrees:
      `shinka-self-play`.
    The handoff is the end of this skill's responsibility ŌĆö the native skill
    owns the task scaffold from there, and `shinka-run` / `shinka-inspect`
    take over for execution and review.

## ShinkaEvolve Retry Behavior

ShinkaEvolve may enter an infinite retry loop on transient errors (LLM
API failures, rate limits, network issues) rather than exiting
immediately. When running downstream Shinka tasks from scan results,
actively monitor log output during smoke tests and evolution runs.
Interrupt (`Ctrl-C`) if the process appears stuck retrying, fix the
underlying issue (wrong API key, quota exhausted, model name typo,
etc.), and re-run.
