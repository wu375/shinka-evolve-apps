# Shinka Figure Evolution Skill

Use this skill to set up a ShinkaEvolve task where the evolving program is a single Python file that generates a rendered research figure. The judge sees only the rendered image and a task-specific rubric — not the candidate source code.

## Composition With Existing Shinka Skills

The core ShinkaEvolve skills live in the `skills/` directory of the ShinkaEvolve repo. If the skills are not already loaded by the agent, first check whether a source clone of ShinkaEvolve exists in the current project or directory tree. If not found, install via `uv pip install shinka-evolve` (prefer `uv` for package management unless the user or project explicitly uses something else).

This skill is a domain-specific **setup layer**. It does not replace the core skills:

- Use `shinka-setup` for generic new Shinka tasks.
- Use `shinka-convert` to adapt an existing plotting script/repo into a Shinka task; you can still hand the converted task to this skill for figure-specific customization.
- Use `shinka-run` once `evaluate.py` and `initial.py` exist — it is the authoritative guide for batches, model availability, and config overrides.
- Use `shinka-inspect` after a run to extract top candidates.

## When To Use

Use this skill when the user says things like:

- "Use Shinka to improve this research figure."
- "Evolve a better diagram for this paper section."
- "Generate figure variants for this writeup."
- "Optimize a matplotlib/SVG figure with an LLM judge."
- "Create a Shinka task whose output is a rendered figure."

Do not use this skill for tasks where the metric is numeric (loss, accuracy, runtime). Use `shinka-setup` instead.

## Minimal Required Inputs

The skill is designed to run from the smallest possible input set:

1. **Research context** — what the paper / project / experiment is about. Can be gathered from conversation, a text file, or extracted from existing documents.
2. **Figure description** — what the figure should communicate (concept diagram, architecture schematic, plot-like prototype, etc.). Can likewise be gathered interactively or from files.
3. **A `.env` file** at the task workspace (or any parent) with API keys for the LLM providers used by Shinka (e.g. `GEMINI_API_KEY`, `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`). The multimodal judge specifically requires `GEMINI_API_KEY`; the proposal LLMs use whichever keys match the models configured in `shinka.yaml`.

Everything else has a default and is **only asked back to the user if they want to override**:

| Input | Default |
| --- | --- |
| Target audience | "Expert scientific reader" |
| Required visual elements | Auto-extracted from the figure description text (LLM extraction at scaffold time, or simple bullet parse if the file already lists them) |
| Data policy | Dummy / placeholder data; figure marked as a visual / conceptual prototype |
| Figure type | Candidates may combine SVG primitives and matplotlib in a single program to create rich figures. Default seed starts with matplotlib; the SVG helpers are always available and can be mixed in. |
| Judge model | `gemini-3-flash-preview` |
| Judge provider | Gemini (multimodal) |
| Proposal LLM models | `gemini-3-flash-preview` only. Confirm with the user before adding other models (e.g. OpenAI, Anthropic) — each requires its own API key and adds cost. |

When scaffolding from just the gathered context and `.env`, populate the rubric and seed using these defaults and **note in `rubric.md` and the final report which defaults were used** so the user can override later without re-running the skill.

## Candidate Program Contract

Every candidate is a single Python file (`initial.py`) exposing:

```python
def make_figure(output_path: str) -> None:
    """Write a renderable figure to output_path."""
```

Output extension must be `.png` (SVG is rendered to PNG before judging).

The evolve region must be tight. Helpers, constants, the `make_figure` signature, and the `run_experiment` wrapper (if present) must live outside `# EVOLVE-BLOCK-START` / `# EVOLVE-BLOCK-END`.

## Data Policy

Because Shinka proposal LLMs cannot dynamically inspect arbitrary disk data during generation, this skill enforces a strict data policy:

1. Prefer a fixed predefined loader inside the seed.
2. Otherwise embed dummy / placeholder data in `initial.py`.
3. Mark the output as a **visual / conceptual prototype** in the rubric and final report. Do not claim numerical data faithfulness from image-only judging.

## Evaluator Contract

Generate `evaluate.py` that:

1. loads environment variables from a `.env` file at the workspace (search current dir + parents), so `GEMINI_API_KEY` is picked up automatically,
2. imports the candidate from `program_path`,
3. calls `make_figure(output_path)`,
4. catches runtime errors and returns clear feedback,
5. validates that a renderable image exists (file exists, nonzero size, parses, opens, not blank),
6. renders SVG → PNG if needed (`cairosvg`),
7. sends only the rendered PNG plus research context + rubric to the multimodal judge,
8. parses the structured judge response,
9. writes Shinka-compatible metrics + correctness outputs.

Required Shinka metrics fields:

- `combined_score` (numeric, higher is better)
- `public` (dict of per-criterion scores visible to the proposal LLM. Important for providing diagnostic feedback to the proposal LLM)
- `private` (dict, metrics that are not visible to the proposal LLM)
- `text_feedback` (string usable for future mutations)

Refer to shinka-setup skill for more details on the metrics fields.

## Rubric Policy

The rubric is task-specific and must be customized.

The rubric will instruct the judge to:

- judge the rendered image,
- flag rendering bugs, unreadable text, clutter, missing concepts,
- flag misleading visual implications,
- judge plausibility against the research context (not source-code quality),
- not assume underlying data is real unless specified.

## Multimodal Judge Integration

Pluggable. The recommended path is the Gemini multimodal API (image bytes + prompt). The evaluator template includes a `judge_image(image_path, rubric_path, context_path)` helper that:

- reads the rendered PNG,
- passes the image plus the rubric and context through the LLM client,
- requests structured JSON output following the rubric's scoring fields,
- pulls credentials from `GEMINI_API_KEY` (loaded from `.env`).

**Client preference order** (the evaluator template should try in this order at scaffold time and pick the first available path):

1. **Prefer ShinkaEvolve's native LLM client** (`shinka.llm`) when it advertises both image input and structured output for the chosen Gemini model. This keeps cost / token tracking consistent with the rest of the evolution run, reuses retry / rate-limit logic, and avoids divergence between proposal-side and evaluator-side LLM calls.
2. **Fallback: call `google-genai` directly** when the native client does not yet support image input or structured output. This is the current default (the multimodal Gemini PR and the structured output PR are not yet merged — see `shinka_apps/pr_designs/multimodal_gemini.md` and `shinka_apps/pr_designs/structured_output.md`).

Detect support by checking whether the native client exposes a multimodal content API (e.g. an image-bearing message variant) and a structured-output / response-schema option. If either is missing, fall back. Once both PRs land, the fallback branch can be removed.

## Workflow

1. Gather the research context and figure description from the user — via conversation, provided files, or a mix.
2. Apply defaults for all unspecified inputs (audience, data policy, judge model). Record which defaults were applied.
3. Create or use the user's task workspace. Ensure a `.env` with the necessary API keys is reachable.
4. Copy `templates/seed_figure_template.py` → `initial.py`. The seed includes both SVG helpers and a matplotlib path; customize both or either to fit the figure requirements. Do not delete one path — candidates may combine SVG elements and matplotlib plots in a single figure.
5. Copy `templates/evaluator_template.py` → `evaluate.py`. The evaluator already loads `.env` and reads `context.md` + `rubric.md` from the workspace.
6. Write the gathered research context to `context.md`.
7. Copy `templates/rubric_template.md` → `rubric.md` and fill in every section using the user's figure description plus defaults. Add a final "Defaults used" section.
8. Copy `templates/shinka_config_template.yaml` → `shinka.yaml` (optional; user may instead use `shinka-run` defaults).
9. Smoke-test and **tune the evaluator judge**. The evaluator is the most
   important part of this workflow — it must be carefully crafted before
   moving on to the actual evolution run.

   ```bash
   python evaluate.py --program_path initial.py --results_dir /tmp/shinka_figure_smoke
   ```

   Confirm: evaluator exits 0, figure exists, rendered PNG exists,
   `metrics.json` has `combined_score`/`public`/`private`/`text_feedback`,
   `correct.json` has `correct`/`error`.

   Then **iterate on the judge prompt and rubric** until scoring is strict
   enough to produce meaningful feedback signals:

   - Run the evaluator several times. Review the `text_feedback` and
     per-criterion scores in `public`. A good judge should identify
     concrete issues (missing elements, clutter, bad contrast) rather than
     giving blanket high scores.
   - If the judge is too lenient (e.g. scores ≥ 8/10 on a clearly rough
     seed figure), tighten the rubric: add explicit deduction rules, raise
     the bar for what counts as "publication-ready", and sharpen the
     failure-mode descriptions.
   - If the judge is too harsh or inconsistent, soften specific criteria or
     clarify ambiguous rubric language.
   - Repeat until the seed figure receives a score that honestly reflects
     its quality and the `text_feedback` gives actionable improvement
     directions. This calibration is essential — a lenient judge produces
     a flat fitness landscape where evolution cannot find a gradient.

10. Hand off to `shinka-run` for evolution batches.
11. After a run, use `shinka-inspect` to extract top candidates and re-render them. Add a short figure-specific report (top images, judge rationale, caveats including the dummy-data label and any defaults used).

## Files

- `SKILL.md` — this file.
- `templates/seed_figure_template.py` — starter `initial.py` with tight evolve block.
- `templates/evaluator_template.py` — starter `evaluate.py` with judge hook and `.env` loading.
- `templates/rubric_template.md` — fill-in rubric (research context, scoring fields, defaults used).
- `templates/shinka_config_template.yaml` — low-budget figure config.

The included files are explicitly **templates**, not final task logic.
